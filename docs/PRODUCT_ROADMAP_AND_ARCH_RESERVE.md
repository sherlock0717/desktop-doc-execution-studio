# 产品路线图与架构预留（案例执行台）

本文档将原「禁止项」调整为**分阶段路线图能力**：在不打断当前「案例执行台」主链的前提下，做产品分层、职责边界与架构预留说明。  
**注意**：不要求一次性实现全部能力；实现时以本文档为契约参考。

---

## 1. 产品路线图分层

| 层级 | 能力 | 为何在此层 |
|------|------|------------|
| **P0 当前核心主链** | 导入材料 → 分类/确认 → 目标模式 → 生成结构化结果包 → 建议项持久化与区块对照工作稿 → Apply 与准备度/对比摘要 | 已形成闭环价值；任何扩展不得破坏该链路的可复跑、可追溯与区块写入语义。 |
| **P0（继续强化）** | 区块对照视图、提交准备度规则、`execution.json` / `draft_apply_summary`、与上次生成对比 | 属于主链的「可完成感」与风险控制，应持续打磨而非旁路。 |
| **P1 下一阶段必须进入设计与部分落地** | **docx 导出**（确定性导出链路）、**refined draft**（整份润色与 current draft 分离）、**案例列表首页**（只读聚合 + 进入执行台）、**多模型切换**（同一提示与 schema 下的 provider/model 选择） | 与「交付物离开本台」「润色不破坏工作稿」「多案例可发现」「成本/质量可调」直接相关；用户心智上已是「产品级」期待，但可分迭代落地。 |
| **P2 更后面的扩展层** | **当前案例内轻聊天辅助**（强约束上下文）、**邮件联动**（投递/提醒）、**多用户/账号体系**（租户、权限、审计） | 依赖稳定的数据模型与外部系统；若过早做会拖垮主链或引入合规成本；适合在 P1 契约稳定后接入。 |

## 2. 各能力职责边界（不泛化）

| 能力 | 职责边界（它负责 / 不负责） |
|------|------------------------------|
| **docx 导出** | **负责**：把「用户已确认要交付的版本」（如某次快照 Markdown 或结构化字段）**确定性**转为 `.docx` 版式与下载；记录导出元数据（时间、来源对象 id）。**不负责**：替用户改内容、替代 Apply、也不默认等于「终稿认可」。 |
| **refined draft（整份润色草稿）** | **负责**：在 **current working draft** 之上生成 **新版本对象**（润色稿），供对比与采纳；可多次生成、保留 generation 指针。**不负责**：不经确认直接覆盖 `resume_working_draft.md`；不替代建议项的逐条 Accept/Apply 流程（除非产品明确定义「一键合并策略」，那也是单独动作）。 |
| **案例列表首页** | **负责**：展示案例集合的**入口信息**（标题、状态、最近活动、准备度摘要），点击进入现有三栏执行台。**不负责**：在执行台外做深度编辑；不复制一套完整材料管理。 |
| **多模型切换** | **负责**：在**同一 JSON schema 与同一渲染链**下切换后端 LLM/模型；记录 `model_name` 与生成元数据。**不负责**：改变业务对象形状、为每个模型维护不同提示分叉（除非显式版本化 `prompt_version`）。 |
| **当前案例内轻聊天辅助** | **负责**：对**当前 case_id + 当前已加载材料/execution 摘要**做短问答（澄清缺口、解释建议、改写提示）。**不负责**：开放域闲聊、多案例切换对话、替代结果包生成主按钮。 |
| **邮件联动** | **负责**：与外部邮箱/通知服务对接（发送导出附件、提醒补材料等），**异步**、可失败重试。**不负责**：在邮件里完成完整编辑闭环；不作为主 UI。 |
| **多用户 / 账号体系** | **负责**：身份、租户隔离、案例与文件的归属与审计日志。**不负责**：替本地单用户场景增加无必要摩擦；实现时可从「单机多 profile」渐进到真多租户。 |

## 3. 架构预留方案（接在哪一层）

| 能力 | 本地确定性逻辑 | LLM 生成层 | 前端交互层 | 外部服务 / 第三方 |
|------|----------------|------------|------------|-------------------|
| docx 导出 | **主**：Markdown/结构化 → docx 模板填充、样式、页眉页脚 | — | 导出按钮、选「导出来源版本」、下载 | 可选：仅上传云存储时不阻塞本地导出 |
| refined draft | 合并策略、diff 结构、**写新文件**路径规则 | **主**：润色 prompt + 输出 `RefinedDraftPayload` | 对比视图、采纳到「新快照」或「合并回工作稿」向导 | — |
| 案例列表首页 | 扫描 `outputs/case_runtime/*/meta.json` 或索引表 | — | **主**：列表页路由 `/cases`，卡片进入 `/case/:id` | 未来：远端同步 |
| 多模型切换 | 校验模型可用、超时 | **主**：`llm_provider` 多后端 + `model_id` | 设置项、展示当前模型 | Ollama/OpenAI 等 |
| 轻聊天 | 注入上下文上限、引用 snippet/case 摘要 | **主**：短对话 completion | 侧栏/抽屉、非全屏 | — |
| 邮件联动 | 队列与重试状态机 | — | 触发点、发送状态展示 | SMTP / SendGrid / Graph API |
| 多用户/账号 | 权限校验中间件 | — | 登录、工作区切换 | OAuth、自建账户服务 |

**目录与对象关系（预留，非实现要求）**

- `outputs/case_runtime/<case_id>/`  
  - 已有：`execution.json`、`resume_working_draft.md`、…  
  - 预留：`exports/`（导出记录 manifest）、`refined/`（`refined_resume_*.md` + payload json）、`case_meta.json`（列表页索引：标题、updated_at、readiness 摘要指针）。
- `app/engine/`  
  - 预留：`export_docx.py`（纯转换）、`refined_draft.py`（调用 LLM + 落盘）、`case_index.py`（列表聚合）；与 `llm_provider.py` 通过统一接口交互。
- `app/frontend/`  
  - 路由预留：`/` 或 `/cases` → 列表；`/workspace` 或保持现状单页后拆 `/case/:id`。
- **接口预留（概念）**  
  - `POST /api/cases/{id}/export/docx` — body：`{ "source": "working_draft" | "refined", "ref_id": "..." }`  
  - `POST /api/cases/{id}/refined-draft/generate` — 返回 `refined_id`，不写覆盖 `resume_working_draft.md`  
  - `POST /api/cases/{id}/refined-draft/{refined_id}/promote` — 可选：显式合并策略  
  - `GET /api/cases` — 列表摘要

## 4. docx 导出链路（优先设计）

| 阶段 | 说明 |
|------|------|
| **输入对象** | `ExportRequest`：`case_id`、`source`（`working_draft` \| `refined`）、`ref_id`（可选）、`template_id`（默认模板）。内容快照：从对应文件或内存快照读取的 Markdown/结构化区块。 |
| **中间对象** | `DocxBuildContext`：解析后的段落列表、样式映射、可选「区块标题」层级；`ExportJobRecord`：job_id、状态、输出路径、校验和。 |
| **输出对象** | `exports/<case_id>/<timestamp>_resume.docx` + `export_manifest.json`（来源、模板版本、sha256）。 |
| **用户确认** | 在点击「导出」前选择导出来源（当前工作稿 / 某次 refined）；下载前可预览纯文本摘要（可选）；**不**自动触发导出。 |
| **避免覆盖 current draft** | 导出**只读**源文件；写入仅限 `exports/` 下新文件；不向 `resume_working_draft.md` 写回。 |

---

## 5. refined draft 链路（优先设计）

| 阶段 | 说明 |
|------|------|
| **输入对象** | `RefineInput`：`case_id`、基准 **`resume_working_draft.md` 全文或结构化 sections**、`goal_mode`、可选「润色强度」、用户勾选「保留事实不编造」。 |
| **中间对象** | `RefinedDraftRecord`：`refined_id`、`parent_generation_id` / `based_on_file_mtime`、`RefinedResumePayload`（与现 schema 可对齐或扩展 `refinement_notes`）、落盘 `refined/refined_resume_<id>.md` + `.payload.json`。 |
| **输出对象** | 新版 Markdown 润色稿 + 元数据；**不**修改 `resume_working_draft.md`。 |
| **用户确认** | 用户在「对照视图」中审阅 refined；**显式**「采纳为本轮投递稿」可：仅标记元数据，或 **复制** 到新的「投递快照」文件；若需写回工作稿，必须单独确认「合并策略」（例如只替换某几节）。默认 **不** 自动覆盖。 |
| **避免覆盖 current draft** | 润色结果**永远**写入独立 `refined_*` 路径；promote/merge 为独立 API 与 UI 步骤。 |

## 6. 聊天辅助最小方案（严格降级）

**定位（必须遵守）**

- 聊天**不能**成为主界面；主界面仍是三栏案例执行台。  
- **只能**围绕**当前 `case_id`**：上下文 = 材料摘要 + `execution` 摘要 + 用户问题；**禁止**回到无案例的通用聊天壳。  
- 入口：**小面板**（右侧抽屉或结果区下方折叠），默认收起；无「新对话」无限滚动替代主流程。

**后续最小形态**

- 单轮或极短多轮（例如最多 5 轮），每轮带 `case_id` 与 `snippet` 引用上限。  
- 输出类型限制为：`explain_gap` | `explain_suggestion` | `rewrite_bullet`（结构化），避免长文淹没工作稿。  
- API：`POST /api/cases/{id}/chat` — 请求体含 `messages[]`，服务端拒绝无 `case_id` 或 case 不匹配。

## 7. 案例列表首页最小方案（仅产品）

**每个案例卡片最少信息**

- 案例标题（或可编辑占位名）  
- 状态：`草稿` / `已生成` / `有未应用建议`（派生）  
- 最近更新时间（材料或 execution 最后写入）  
- 一行准备度：**绿/黄** + 简短文案（如「缺 JD」「3 条建议待应用」）

**体现准备度 / 时间 / 缺口**

- **准备度**：复用 `readiness_summary` 规则产出的布尔组合，映射为 **单一主标签**（不要堆四个 chip）。  
- **时间**：`max(材料 mtime, execution.last_updated)`。  
- **缺口数**：`len(missing_inputs)` 或聚合为「N 项待补」。

**避免页面变杂**

- 列表页**无**三栏、无全文编辑；仅 **一行操作**：进入执行台。  
- 筛选器最多 1～2 个（如「仅显示待处理」）。  
- 与「执行台」信息架构分离：列表 = **导航**，执行台 = **操作**。

## 8. 设计师视角 / 用户视角再审

**设计师视角**  
分层后 P0 主链保持一条视觉与任务主线；P1 能力以「导出」「润色稿」「列表入口」等**独立入口与独立对象**出现，避免与区块对照抢焦点。聊天若出现，只能是低频、低视觉权重的辅助层。列表页用一张卡片一个焦点（标题 + 状态 + 一条摘要），避免仪表盘化。

**用户视角**  
用户最怕「润色把稿子弄没」—— refined 与 working draft 分离、合并需确认，能直接降低不信任感。导出 docx 是「拿走」的动作，应在心智上与「保存工作稿」区分。多模型是「高级设置」，默认隐藏或放在生成前一步。聊天若只能问当前案例，反而比通用助手更安心。
